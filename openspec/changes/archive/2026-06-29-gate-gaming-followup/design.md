# Design — Gate-Gaming Follow-up

## Architecture

Three independent prose/script edits to skills we own, plus their content tests. No hook, config,
or routing changes. Stacks on `feature/gate-gaming-detection`.

### Empirical basis (the load-bearing part)

Instrument: labeled corpus (26) + two **blind held-out** sets (20, 21) generated by *other* agents
from the abstract gate-gaming definition, benchmarked with an independent harness (precision/recall/F1
over scored cases; ambiguous reported separately).

| Detector | Corpus F1 | Held-out-1 F1 | Held-out-2 F1 |
|---|---|---|---|
| v1 (shipped: removed-assert OR added-skip) | 0.79 | 0.44 | 0.64 |
| v3 (count-delta + tautology + strong→weak) | 0.93 | **0.14** | — (rejected) |
| v4 (v1 + comment-exclusion + skip-dialect breadth) | — | 0.70 | **0.64 (tie)** |

**Reading:** v3 overfit (in-sample 0.93 → held-out 0.14, below baseline). v4's held-out-1 gain did
not reproduce on held-out-2 (NUnit `[Ignore]`, RSpec `xit '...'`, docstring-muted markers all slipped
its broadened-but-still-incomplete regex). Two blind rounds ⇒ regex breadth is whack-a-mole; the
ceiling is structural (line diffs cannot see subject-stubs, control-flow, block/docstring muting, or
move-vs-delete). **No detector code ships.**

### Edit 1 — project-verification Limits note
Trust-calibration prose after the gate-gaming capture block: `clean` ≠ proof; state the measured F1
range and the blind spots + false-alarm classes; point at the future ceiling-breaker. Advisory framing
preserved (the detector remains advisory, not a trust boundary).

### Edit 2 — batch-scripting per-file postcondition
Step 3 gates `OK` on `[ -s tmp ] && ! cmp -s tmp bak && sanity_check tmp` (non-empty, changed,
parses) instead of the `claude -p` exit code alone. `sanity_check` is a signposted per-file-type
probe the user fills (py_compile / node --check / jq / structural token). Failures log `FAIL` and
enter the existing retry path — never a silent `OK`. Minimal form; retry-classification stays killed.

### Edit 3 — outcome-review failure-cause split
A required Cause for any non-`Confirmed` hypothesis: instrumentation-broken (fix tracking, re-measure)
/ adoption-gap (increase rollout, re-measure) / product-miss (iterate or roll back) / inconclusive-data
(extend window). Recommendation must follow from cause.

## Trade-offs

- **Honesty over a marginal detector tweak.** v4 never regressed (Pareto-safe) but didn't robustly
  improve; shipping it as an "improvement" would overstate it. The validated, generalizing win is the
  limits note — it directly counters the false-confidence failure mode.
- **Postcondition is advisory and per-type.** `sanity_check` is left to the operator (file types vary);
  the non-empty + changed checks are universal and cheap.

## Dissenting views / decisions

- Considered shipping v4's comment-exclusion + dialect breadth as a Pareto-safe refinement. **Rejected
  for now**: held-out-2 showed no net benefit, and chasing dialects is whack-a-mole. Recorded so it is
  not silently re-attempted without the held-out protocol.
- Escalation path (coverage-delta / LLM-judge) deferred, not built — it needs its own evaluation and
  the same blind-held-out gate.

## Eval strategy

Deterministic — content-assertion tests per skill (TDD-style presence checks on the load-bearing
strings). The detector itself is unchanged, so its behavioral fixtures are untouched. The empirical
benchmark that drove these decisions lives in session research artifacts, not the repo test suite.
