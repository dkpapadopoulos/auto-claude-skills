### Fixed
- **`runtime-validation` CLI scenario loop no longer `eval`s eval-pack command strings** (`skills/runtime-validation/SKILL.md`, guards in `tests/test-adversarial-governance.sh`). The documented CLI execution snippet used `eval "${cmd}"` on `.inputs.command` read from eval-pack JSON — a shell-injection vector that also runs in the *current* shell (so a crafted fixture could mutate the validation environment) if an eval pack is ever attacker-influenced. Replaced with a structured `.inputs.argv` array executed with **no shell** (preferred, zero injection surface) and a child-subshell `bash -c` fallback for the legacy string form, plus an explicit trust-boundary note (eval packs are TRUSTED committed fixtures only). Surfaced by the PR-touched-file checklist; pre-existing. TDD: `assert_not_contains 'eval "${cmd}"'` + trust-note assertion added RED then GREEN. The `agent-safety-review` description gloss was also tightened (dropped "i.e. potential … exposure", kept the "lethal trifecta" keyword) for catalog density. (PR #63)

- **`scripts/scenario-coverage.sh` over-flagged documentation/guidance capabilities** (the report shipped in the prior commit; regression fixtures `doc-cap` + `heading-noise-cap` in `tests/test-scenario-coverage.sh`). The probabilistic-`THEN` count scanned every spec line, so it counted matches inside `### Requirement:` / `#### Scenario:` headings and `Given`/`When` preconditions, and treated static-artifact assertions (the doc/banner/`.md`/config-flag content a grep test already covers) as model behavior. Result: doc/guidance capabilities like `unified-context-stack` showed 31 "uncovered" probabilistic clauses when the genuine behavioral count is ~0. Fix (TDD): count probabilistic verbs only in OUTCOME lines — drop `^#` headings and `GIVEN`/`WHEN` precondition lines, and subtract a static-artifact negative filter (`banner|phase doc|tier doc|the doc|\.md|appears in|MUST mention|contain|name|assert|context-capability|flag MUST|...`). `unified-context-stack` now correctly clears (31→1, below threshold); `alert-hygiene` correctly stays flagged (5 genuine behavioral outcomes, no pack — the report's one true gap); the backfilled `security-scanner`/`incident-trend-analyzer` packs stay `has-pack`; counts are more honest (`incident-analysis` 32→11). The two fixtures isolate the two distinct fixes (artifact-subject vs heading/GIVEN noise) and were both RED-verified before the fix. Capability: `behavioral-evaluation`.
- **Session-token resolution is now payload-first — concurrent-session singleton race fixed** (new `hooks/lib/session-token.sh`; converted readers `hooks/openspec-guard.sh`, `hooks/skill-activation-hook.sh`, `hooks/skill-completion-hook.sh`, `hooks/consolidation-stop.sh`, `hooks/compact-recovery-hook.sh`; writer `hooks/session-start-hook.sh` sources the lib; new regression `tests/test-session-token-race.sh`, 12 assertions incl. two interleaved simulated sessions). `~/.claude/.skill-session-token` is a shared singleton with last-writer-wins semantics: with ≥2 concurrent sessions, hooks that resolved "my token" by reading it back evaluated ANOTHER session's composition state — observed live in the PR #50 session (token rotated 3× post-#47; 17 claude processes) as the push gate denying a legitimate push from a foreign incomplete chain (issue #51). The two prior fixes (#43 reuse-window, #47 transcript keying) solved *derivation instability*; this is *pointer contention*, which they cannot address. Fix: hooks derive the token from their own stdin payload's `transcript_path` (`session-<basename>`, format single-sourced in the new lib) and use the singleton only as fallback when the payload lacks the field or jq is missing; `compact-recovery-hook.sh` now drains stdin BEFORE token use (it previously read the singleton at the top and stdin only at the bottom); the activation hook re-stamps the singleton with its payload-resolved token each routed prompt, narrowing the residual window for no-payload SKILL.md consumers (`openspec-ship`, `product-discovery`, `implementation-drift-check`, `runtime-validation` — out of scope, revival trigger documented) to one prompt-width. jq fork budget held: guard and activation batch `transcript_path` into their existing jq call (`\x1f`-joined, transcript first since a path cannot contain `\x1f`); the completion hook's three extractions merge into one (net fork count DOWN by one). Resolution deliberately never prefers "whichever token has state" — that would reintroduce the race. Capability: `skill-routing`.

Fix: when the chain is unchanged, the write unions the computed prefix with the on-disk `.completed` (chain-ordered, deduplicated, out-of-chain entries dropped — union, not a length floor, so skipped steps are never fabricated as done). Chain switch, pure-cancel, and token rotation remain the only resets; fail-open on missing/malformed prior state (and on jq <1.5, where `--slurpfile` is unavailable, degrading to the prefix-only write); `current_index` stays the display anchor. Every fix empirically pre-verified under `/bin/bash` 3.2 `[[ =~ ]]` (bug proven on old regex, blocked on new, inflections preserved): `review`→pREVIEW (requesting-code-review, agent-team-review, pr-review hint), `ship`→relationSHIP + `tag`→sTAGing/unTAGged + `merge`→eMERGEd/eMERGEncy + `release`→preRELEASE + `complete`→inCOMPLETE + `finish`→unFINISHed + `ready.to`→alREADY-TOo (verification-before-completion), `hang`→cHANGes/cHANGelog/excHANGe (systematic-debugging — a priority-50 process skill firing on a large fraction of dev prompts), `nits?`→uNIT/iNITialize/moNIToring/defiNITion (receiving-code-review), `as.?built`→wAS-BUILT (openspec-ship), `set.?up`→asSET-UPload/offSET-UPdate (brainstorming), `mass`→MASSive (batch-scripting, the one both-side anchor), `continue`→disCONTINUEd (executing-plans), `release` (deploy-gate), `merge` (github-mcp hint). - **Design-guard heading greps were exact-match and missed real-world variants** (`hooks/skill-activation-hook.sh` PLAN-phase DESIGN COMPLETENESS block, regressions in `tests/test-routing.sh`). The three section checks (`^## Capabilities Affected`, `^## Out-of-Scope`, `^## Acceptance Scenarios`) scored an approved design doc using `## Out of Scope` (spaces) as missing the section — surfaced by the format-handoff eval (PR #48), where the tolerant-regex alternative matched the full mutation set at zero convention cost and beat the re-paused YAML front-matter proposal. Fix: case-insensitive ERE (`grep -Eiq '^#{2,3} .*out[- ]of[- ]scope'` shape) tolerant of h2/h3 level, space-or-hyphen joins, and prefix/suffix text (emoji, "& Non-Goals"). h4+ headings, body-text mentions, and leading whitespace before `##` intentionally still do not count (negative regression test). Verified under BSD grep incl. `LC_ALL=C` multibyte; advisory block only — `hooks/openspec-guard.sh` deny boundary untouched. Capability: `skill-routing`.
- **Behavioral eval runner broken under current Claude CLI — variadic `--disallowedTools` swallowed the prompt** (`tests/run-behavioral-evals.sh`, regression guard in `tests/test-run-behavioral-evals-variance.sh`). The current CLI parses `--disallowedTools <tools...>` as variadic, so the trailing prompt positional was consumed as tool names ("Permission deny rule ... matches no known tool") and every inner run died with "Input must be provided" — all eval packs were unrunnable under the current CLI. Fix: the constructed prompt is delivered via stdin (`printf '%s' "$PROMPT" | claude -p ...`); the tool list is comma-joined into a single token. The test stub now reads stdin with a bounded `read -t 5` and emits a non-matching result when no prompt arrives, so an argv regression fails the suite loudly instead of deadlocking (verified by positive + negative controls). Related environment note: `claude -p --bare` currently fails OAuth ("Not logged in") in nested sessions — the format-eval driver defaults to no-bare with a dated comment. Capability: `behavioral-evaluation`.
- **Variance report bled assertion regexes into the Description column** (`tests/run-behavioral-evals.sh` `write_variance_report`/`update_counter`, regression in `tests/test-run-behavioral-evals-variance.sh`). The counter store joined assertion text and description with `|`, then split on the first `|` when rendering — so any `text` assertion whose regex contained `|` (e.g. the model-routing catch-detector alternation) leaked the regex into the report. Counting was always correct; only the rendered Description was garbled. Fix: store text and description as separate tab fields. Capability: `behavioral-evaluation`.
- **Session-token rotation on SessionStart re-fires** (`hooks/session-start-hook.sh:62-91`, regression tests in `tests/test-session-token.sh` + new `tests/test-session-token-reuse.sh`). The hook's fallback path (used when `session_id` is absent from stdin, jq is missing, or stdin is a TTY) used to unconditionally generate a fresh `<epoch>-<pid>-<rand>` token, rotating the value on every SessionStart re-fire. ScheduleWakeup-driven re-invocations hit this path and orphaned in-flight composition state — manifesting as the openspec-guard push gate blocking pushes after a long sleep because `requesting-code-review` / `verification-before-completion` lookups returned a fresh empty chain. Fix: in the fallback path, reuse the existing token if `~/.claude/.skill-session-token` was modified within `SKILL_TOKEN_REUSE_WINDOW` seconds (default 14400 = 4 hours). Long-stale tokens still rotate, preserving the original collision-defense across genuinely distinct sessions. Existing `test-session-token.sh` updated to reflect the new contract (sequential re-fires within the window MUST reuse; stale tokens MUST rotate). Capability: `skill-routing`.
- **Session-token rotation on resume/compact** (`hooks/session-start-hook.sh`, new regression test `tests/test-session-token-resume.sh`). The token-stability fix above keyed the token off stdin `session_id`, assuming it is stable for the life of a conversation. It is not: `resume` and `compact` deliver a fresh `session_id` while the same conversation continues (this hook's empty matcher fires on both), rotating the shared `~/.claude/.skill-session-token` and orphaning every composition/openspec-state file keyed off it — producing false-negative openspec-guard push-gate and consolidation-guard blocks even when REVIEW/VERIFY/consolidation genuinely ran. Root-caused 2026-06-05: the reuse-window fallback (#43) never ran because it only guards the no-`session_id` path. Fix: derive the token from a conversation-stable id — the `transcript_path` basename, which persists across resume/compact — before falling back to `session_id`, then to the reuse-window/random logic. `transcript_path` is already consumed by `compact-recovery-hook.sh`/`pre-compact-hook.sh`, confirming the harness delivers it. Capability: `skill-routing`.
- **Stale per-session state files accumulated unbounded** (`hooks/session-start-hook.sh`, new regression test `tests/test-state-file-cleanup.sh`). Session start pruned `.skill-prompt-count-*` and `.skill-zero-match-count` but never `.skill-composition-state-*` / `.skill-openspec-state-*`, so they piled up (hundreds observed — amplified by the rotation bug above). Fix: age-based prune at session start of composition/openspec state files untouched for longer than `SKILL_STATE_RETENTION` seconds (default 604800 = 7 days), with an explicit skip for the current session token so a long-idle-then-resumed conversation is never cut. `learn-baselines` are intentionally persistent and are not touched. Fail-open (stat/rm failures never abort the hook). Capability: `skill-routing`.
